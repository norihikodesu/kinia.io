# Kinia ランディングページ改善指示

このファイルは、`https://kinia.io/` の現状確認と `D:\Clode作業フォルダ\アプリ開発\kinia-website\index.html` の内容を踏まえて作成した、Claude Code向けの改善指示です。  
小手先の調整ではなく、構成・読みやすさ・モバイル体験・信頼感をまとめて引き上げる前提で対応してください。

## 0. 前提

- 現状のコアメッセージ「監視のない見守りを」は良いので残す。
- ダークトーン自体は悪くない。既存のブランド感は維持してよい。
- ただし現状は「文章量が多い」「カードが多い」「最初の数画面でサービス理解が進みにくい」「スマホでの情報密度が不均一」という問題がある。
- 一番大事なのは、**最初の2画面以内で以下を伝えること**:
  1. これは何のアプリか
  2. 既存の位置共有アプリと何が違うか
  3. 今ユーザーは何をすればいいか

## 1. 現状の主な問題点

### 1-1. ヒーローで「思想」は伝わるが「プロダクト」が見えにくい

- 現状のヒーローはコピー自体は良いが、アプリアイコン中心で、実際に何ができるかが一目で入ってこない。
- `Google Playで公開準備中` が押せないCTAになっており、主CTAとして弱い。
- ベータ導線が副次CTAに落ちているため、公開前LPとしての行動導線が弱い。

### 1-2. ストーリーが長く、価値理解より先に読む負荷が来る

- 「なぜ作ったか」は良いが、LP前半でこの分量を読むのは重い。
- スマホでは、ヒーローのあとに長文が続くため、プロダクトの要点にたどり着く前に離脱しやすい。

### 1-3. カード形式の使いすぎで、ページ全体のリズムが単調

- カード自体が悪いのではなく、**ほぼ全部の情報をカードに載せていること**が問題。
- `Problem`、`Use cases`、`Security`、`More Features` で同じような箱が続くため、見た目の変化が少なく、読み進める気持ちよさが弱い。
- しかも一部カードは文章量が多く、カードの「一覧で素早く理解できる」利点を殺している。

### 1-4. スマホで読んだ時の導線と密度が最適化されていない

- ナビにリンクが多く、モバイル幅で窮屈になりやすい。
- 共有レベルの表はスマホで圧縮されると読みにくい。
- セクション余白が大きく、長文も多いため、縦スクロール量がかなり長い。

### 1-5. 信頼面で未完成に見える箇所がある

- `お問い合わせフォーム` が `https://forms.gle/PLACEHOLDER` のままなのは公開LPとしてかなり危険。最優先で解消すること。
- 画像の一部が「AIで作った汎用ビジュアル」に見えやすく、プロダクトの具体感より装飾感が先に立つ箇所がある。
- `ui_member_list.webp` は英語UIになっていて、日本語LPとの接続が弱い。実アプリらしさも不足している。

## 2. カード形式についての判断

### 結論

- **カード形式は悪くない。**
- ただし、**比較しやすい情報・並列に眺める情報にだけ使うべき**。
- 今回のLPでは、カードの数を減らし、役割を絞るべき。

### カードを使ってよい場所

- 3つの価値訴求
- ユースケース
- セキュリティの要点

### カードをやめた方がよい場所

- 長めの問題提起
- ストーリー本文
- 詳細説明が必要な思想や設計思想

### 運用ルール

- 1枚のカードは「1メッセージ + 補足1文」程度に収める。
- 1カード内に3文以上入るなら、カードではなく通常セクション化を検討する。
- 連続するセクションすべてをカードグリッドにしない。  
  少なくとも「カード群」の間に、以下のどれかを挟むこと:
  - 左右分割のテキスト+画像セクション
  - 大きな引用ブロック
  - 縦方向のステップ説明
  - 比較行ベースの帯セクション

## 3. 必須の構成変更

ページ順を次のように組み替えてください。

1. ヒーロー
2. Kiniaの違いを3点で即説明
3. どう通知されるか / 何が見えるか
4. ユースケース
5. フェアであること / 監視しない思想
6. セキュリティ要点
7. 開発のきっかけ（今の長いストーリーを短く再編集）
8. 将来構想
9. 最終CTA
10. お問い合わせ

### 補足

- 今の `Story` セクションは削除しなくてよいが、**後半に移動して3割以上圧縮**すること。
- `Name` セクションは優先度が低い。残すならフッター直前の短い補足に落とす。  
  迷うなら一旦削ってよい。
- `More Features` は単独セクションとして大きく見せすぎない。  
  上位の価値訴求に統合するか、コンパクトに再編する。

## 4. ヒーローの具体指示

- 現状の「アプリアイコンを大きく置く」構成はやめる。
- ヒーローは **左にコピー、右にアプリ画面のビジュアル** の構成を基本にする。モバイルでは縦積み。
- 右側は可能なら**実アプリのスクリーンショットをベースにしたモック**を使う。難しければ、現状より実在感のある再生成画像に差し替える。

### ヒーローに入れるべき要素

- 強い見出し: `監視のない見守りを`
- 1文の補足: `家族の居場所を追いかけず、必要な通知だけでつながる位置共有アプリ`
- 3つの短い要点
  - `通知エリアだけ共有`
  - `全員フェア`
  - `運営も中身を見られない設計`
- 主CTAは押せるものに変更
  - 第1候補: `ベータテスターに応募する`
  - 第2候補: `公開時に通知を受け取る`
- `Google Playで公開準備中` はボタンではなく、ステータスバッジや補足文に下げる

## 5. フォントと読みやすさの必須条件

このLPは、誰でも読める大きめのフォントを前提に再設計してください。

### ルール

- モバイル本文は **18px前後を最低ライン** にする
- 小さいラベルでも 12px 未満にしない
- 行間は狭くしすぎない。本文は `1.75` から `1.95` の範囲で調整
- 1段落が長くなりすぎる場合は改行・段落分割を増やす
- 1セクション内の段落数を減らし、1ブロックごとの意味を明確にする

### 現状から特に直したい点

- ナビ文字が小さい
- セクションラベルが小さすぎて補助情報としても弱い
- フッターと問い合わせ周辺の文字が小さい
- 長文段落が連続するとスマホで疲れる

## 6. スマホ縦スクロール最適化の必須条件

- モバイルファーストで再設計する
- 最初の2〜3スクリーンで、サービス理解とCTA到達ができるようにする
- 横スクロールは絶対に出さない
- 表組みはモバイル専用表現に置き換える
- ナビはモバイルで整理する

### 具体対応

- ナビは次のどちらかにする
  - ハンバーガーメニュー
  - 表示リンクを3つ程度までに絞る
- `2つの共有レベル` の表は、モバイルでは以下のどちらかに変える
  - 比較カード2枚
  - 項目ごとの縦並び比較リスト
- セクション余白は `100px固定` ではなく `clamp()` で調整し、モバイルでは詰める
- 長いストーリーやセキュリティ説明は、本文を圧縮して視線移動を短くする

## 7. セクション別の改善指示

### 7-1. 「既存の位置共有アプリが抱える問題」

- 4枚カードのままでもよいが、文章量を減らすこと
- 各カードは1テーマ1メッセージに絞る
- このセクションの後半にある長文補足は短くする
- このセクション単独で重くしない

### 7-2. 「Kiniaの原則」

- ここはカードにせず、今のような文章ベースでよい
- ただし引用・太字・短い見出しを使って、読みのリズムを作る
- 文章量は今より少し削る

### 7-3. 「全世代に向けたつながりの形」

- このセクションはかなり重要なので残す
- ただし、通知例ブロックは見た目をよりプロダクトっぽくする
- 可能ならここで「何が相手に見えて、何が見えないか」を1目で分かるようにする

### 7-4. 「Kiniaの答え」

- 現状の方向性は良い
- ただし右の画像をもっと実在感のあるものに差し替える
- 箇条書きは今のままでよいが、見出しと補足文のメリハリを強める

### 7-5. 「あなたの家族に合った使い方」

- カード利用は継続してよい
- ただし4枚とも同じ見た目・同じ密度だと平坦なので、アイコンサイズ、余白、見出し強弱を整える
- 1カードあたりの本文は短くする

### 7-6. 「2つの共有レベル」

- デスクトップでは表でもよい
- モバイルでは別UIに切り替える
- ここは「比較しやすさ」が最優先で、見た目の装飾より情報整理を優先する

### 7-7. 「セキュリティ」

- 内容は強いが、現状は長い
- まず最初に3行で要約する
  - `サーバーに保存される前に暗号化`
  - `合言葉はサーバーに送られない`
  - `運営でも中身を読めない`
- その下に3〜4個の根拠カードを置く構成にする
- 長文説明はFAQや補足に逃がせるなら逃がす

### 7-8. 「開発のきっかけ」

- 今の長い `Story` をここへ移す
- 全部は載せず、重要な段落だけ残して短くする
- ここは感情移入パートとして残してよいが、前半に置かない

### 7-9. CTA / Contact

- `お問い合わせフォーム` の `PLACEHOLDER` は公開状態として不適切。最優先で修正
- 実フォームURLが未確定なら、少なくともメールCTAを主導線にして、死んだリンクは消す
- 最終CTAは `公開準備中` ではなく、今できる行動を明示する

## 8. デザインのトーン指示

- 今のダークテーマは維持してよい
- ただし、全セクションが同じ濃い箱で並ぶ感じは減らす
- 背景の濃淡や余白でリズムを作る
- 「強いコントラストのカードを大量に置く」のではなく、
  - 素の背景
  - 薄い面
  - 強調面
  を切り替える

### 避けたいこと

- 何でもカード化する
- 装飾のためだけの光彩やグローを増やす
- 情報量を減らさず見た目だけ派手にする
- 文章をほぼそのまま残してデザインだけ変える

## 9. 画像・アイコンの改善方針

### 結論

- 画像は全交換までは不要
- ただし以下は再生成または差し替え推奨:
  - `ui_member_list.webp`
  - ユースケース4枚の小アイコン群

### 理由

- `ui_member_list.webp` が英語UIで、日本語LPとトーンがずれる
- 「テキストベースがメイン」という重要差別化ポイントなのに、現状の見せ方がやや汎用的
- 小アイコン群は雰囲気は合っているが、ややAI生成感が強く、プロダクトの信頼感より装飾感が出やすい

## 10. Gemini に渡す指示

必要に応じて、以下をGeminiに依頼してください。  
生成後はClaude側で、サイズ最適化・差し替え・トーン統一まで行ってください。

### 10-1. `ui_member_list.webp` 再生成

**狙い**: 英語UIではなく、日本語LPに自然につながる、より実在感のあるモックにする

```text
Create a smartphone mockup for a Japanese family safety app landing page.
The UI must feel like a real product screen, not generic concept art.

Requirements:
- Dark theme UI
- Japanese labels, not English
- Text-first member list, not a map
- Show 4 family members
- Each row should include:
  - display name in Japanese (e.g. お母さん, 太郎)
  - last update timing (e.g. 5分前)
  - current shared area only (e.g. 自宅, 学校, 会社)
  - a small subtle status indicator
- No chat bubbles
- No map pins
- No exaggerated 3D effects
- Clean spacing, high readability, product-like realism
- Portrait smartphone mockup, centered
- Background should match Kinia dark landing page tone (#12121F)
- Accent colors: light blue, warm orange, soft violet
- No text outside the phone
- Make it feel trustworthy, calm, and minimal
```

### 10-2. ユースケース用アイコン4点の再生成

**狙い**: 汎用AIイラスト感を弱め、統一感のあるシンプルな視覚言語にする

対象:

- `icon_child_watch.webp`
- `icon_distant_parent.webp`
- `icon_meetup.webp`
- `icon_share_all.webp`

```text
Create a cohesive set of 4 square illustrations for a Japanese family safety app landing page.
They should look like a unified icon family, not unrelated AI artworks.

Style requirements:
- Minimal flat illustration
- Dark background
- Soft accent colors only
- Very clean silhouettes and shapes
- Low detail
- Calm, warm, trustworthy
- No text
- No glossy 3D rendering
- Consistent stroke weight and lighting across all 4 images

Illustration themes:
1. Child commuting safely between home and school
2. Adult checking on parents living separately, with low psychological pressure
3. Temporary location sharing for meeting up
4. A family choosing to share everything voluntarily
```

### 10-3. ヒーロー用ビジュアルが必要なら追加生成

実アプリ画面を使えない場合のみ、以下を依頼してください。

```text
Create a hero visual for a landing page of a Japanese family location-sharing app.
The concept is "gentle connection without surveillance."

Show:
- A smartphone with a clean text-based family status UI
- A soft sense of family connection
- No map-heavy interface
- No surveillance cameras
- No overly dramatic effects

Style:
- Calm, premium, minimal
- Dark background
- High readability
- Landing-page ready
- Trustworthy rather than flashy
```

## 11. 実装時の判断ルール

- 「良い文章だから残す」ではなく、「LPとして読み進めやすいか」で判断する
- 文章は削る勇気を持つ
- 1セクション1役割を徹底する
- モバイルで読みにくい表現は、デスクトップで多少きれいでも採用しない
- カードは必要最小限にする

## 12. 完了条件

以下を満たしたら完了です。

- ヒーローでサービス内容・違い・行動導線が即分かる
- 本文の読みやすさが上がり、スマホでも負荷が低い
- カードの数と密度が整理され、ページ全体の見た目に緩急がある
- `お問い合わせ` が未完成リンクでなくなる
- `ui_member_list.webp` を中心に、ビジュアルの信頼感が上がる
- モバイルで比較表が読める
- 「監視のない見守り」という思想が、今より短い導線で理解できる
